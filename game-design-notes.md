## 1. Coyote Time / Input Buffer (관용성 디자인)

_플레이어 입력 타이밍에 작은 허용 윈도우를 두어, 인지-실행 사이의 간극을 시스템이 흡수하는 디자인 원리._

**개념**
인간의 인지-운동 반응에는 약 100~250ms 지연이 있다. 게임이 "버튼을 누른 순간"을 글자 그대로 받으면, 플레이어 의도는 "절벽 끝에서 점프"였는데 결과는 "허공에서 점프 → 실패"가 된다. 이건 어려움보다 **불공정**하게 느껴진다.

짧은 시간 윈도우로 입력 의도를 보존:
- **Coyote Time (사후 관용)** — 절벽에서 떨어진 후 ~0.1s 동안 점프 허용. 명칭은 Looney Tunes의 와일 E. 코요테가 절벽을 지나 허공을 달리다 뒤늦게 추락하는 연출에서 유래
- **Input Buffer / Jump Buffer (사전 관용)** — 착지 직전 ~0.1s 점프 입력 저장 → 착지 즉시 발동. 격투 게임의 콤보 입력 버퍼링과 동일한 원리 — 선입력을 기억해 조건 성립 순간 즉시 실행
- **Hit Buffer / Cancel Buffer** — 격투 게임의 가드/필살기 입력 윈도우

**핵심 원리**
**난이도와 관용성은 별개 축이다.** 어려운 게임도 관용적일 수 있고(Celeste), 쉬운 게임도 빡빡할 수 있다. 관용성은 "플레이어 의도를 시스템이 얼마나 추론하는가"의 문제.

관용성이 낮으면 "내 잘못이 아닌데 실패한" 경험이 누적 → 리텐션 ↓. 너무 관대하면 입력 정밀도가 게임플레이 핵심인 장르(리듬, 격투)의 깊이 ↓.

**설계 판단 기준**
- 적합: 플랫포머, 액션 — 물리 타이밍 정확도보다 플레이어 의도 달성이 우선
- 비적합: 리듬 게임, 격투 게임의 파훼 입력 — 정밀도 자체가 숙련도 측정 수단. 윈도우를 너무 넓히면 게임 깊이 훼손

대표 사례:
- **Celeste** — Coyote, Jump Buffer, Grab Buffer 모두 적용. 어렵지만 "내 잘못" 느낌 강함
- **Super Mario Bros** — 1985년부터 Jump Buffer 존재 (의도하지 않은 결과로 시작)
- **Street Fighter** — 입력 윈도우 2~3프레임이 캐릭터 숙련도의 일부

## 2. Hitscan 판정 기준 — Camera vs Muzzle Origin

_FPS 사격 판정을 카메라(눈)에서 할지 총구에서 할지의 설계 결정. 직관적 조작감과 물리적 공정성 사이의 트레이드오프._

**개념**
Hitscan은 실제 총알 오브젝트 없이 발사 방향으로 레이(Ray)를 쏴 충돌 여부를 즉시 판정하는 방식. 레인보우 식스 시즈, 오버워치 등 주요 FPS에서 표준으로 쓰인다.

**핵심 원리**
핵심 설계 결정: **레이를 어디서 쏠 것인가**

| 기준 | 직관성 | 공정성 | 주요 문제 |
|---|---|---|---|
| 카메라 (눈) | 크로스헤어 = 탄착점 → 직관적 | 총구가 가려져도 명중 가능 | 헤드 글리치 |
| 총구 (Muzzle) | 조준선과 탄착점 오차 발생 (근거리일수록 심함) | 총구가 막히면 명중 불가 → 공정 | 직관성 ↓ |

**헤드 글리치 (Head Glitch)**
카메라 기준에서 발생하는 구조적 설계 결함. 캐릭터 총구가 엄폐물에 완전히 가려진 상태에서도 플레이어 카메라에 적이 보이기만 하면 레이가 엄폐물을 통과해 명중 판정이 난다. 반면 상대방은 엄폐물에 가려진 플레이어를 물리적으로 맞출 수 없어 비대칭 우위가 발생.

**실무 해결책 — Hybrid 이중 검사**
카메라 기준 판정을 유지하되 총구 가시선 체크를 추가로 수행:
1. 카메라에서 1차 레이 → 명중 후보 좌표 확정
2. 총구에서 명중 후보 방향으로 2차 레이 → 총구가 엄폐물에 막히면 판정 취소

직관성(카메라 기준)과 공정성(총구 기준) 중 하나를 택하는 것이 아니라, *두 조건을 독립적으로 검사해 교집합에서만 판정*하는 구조가 실질적 해결책. [[game-design-notes]] #1 관용성 디자인과 같은 맥락 — 직관적 입력을 시스템이 수용하되, 의도치 않은 이득은 예외 처리로 차단.

**설계 판단 기준**
- 적합: 싱글플레이어 FPS — 직관성 우선, 헤드 글리치 예외 처리로 보완
- 비적합 (순수 카메라): 경쟁 PvP FPS — 엄폐 불공정이 게임 신뢰를 훼손. Hybrid 이중 검사 또는 총구 기준으로 보완 필요

대표 사례:
- **오버워치** — 카메라 기준 + 총구 가시선 Hybrid
- **레인보우 식스 시즈** — 엄폐 시스템 복잡성으로 카메라/총구 오차가 두드러짐
- **Valorant** — 이동 패널티, 크라우칭 스프레드 등 추가 레이어로 보완

## 3. 절차적 맵 생성 (Procedural Generation)

_완전 무작위 대신 개발자가 설계한 구조 위에 무작위 요소를 결합해 매번 새로운 맵을 만드는 방식. 리플레이어빌리티와 난이도 균형 사이의 트레이드오프를 "레이어 분리"로 해결하는 설계 패턴._

**개념**
게임이 실행될 때마다 알고리즘이 새로운 맵을 생성하는 방식. 매번 다른 경험을 주어 리플레이어빌리티가 높아지지만, 완전 무작위는 클리어 불가 구조(탈출구 없음, 진행 막힘)를 낼 수 있다.

**핵심 원리**
해결책은 두 레이어를 분리하는 것:
- **보장 레이어** (수제작): 클리어 가능성, 진행 구조, 특수 방 배치 조건 — 개발자가 규칙으로 고정
- **다양성 레이어** (무작위): 템플릿 선택, 타일 할당, 확장 방향 — 씨드마다 다르게 결정

**3가지 접근 방식**

**① 바둑판 + 템플릿 방식 (Spelunky)**
스테이지를 N×N 격자로 나누고, 먼저 입구→출구 메인 경로를 무작위 결정(보장 레이어). 각 칸은 연결 방향에 맞는 수제작 템플릿 풀에서 하나를 선택해 배치. 이후 좌우 반전·장애물 배치로 변형(다양성 레이어).
- 탈출구가 항상 존재하는 경로 보장 → 클리어 불가 맵 차단
- 템플릿 × 반전 × 장애물 조합으로 시각적 다양성

**② 고정 지역 + 타일 세트 방식 (Dead Cells)**
지역(바이옴) 단위로 입출구·특수 방을 먼저 배치(보장 레이어). 나머지 일반 방에 수제작 타일 풀에서 무작위 할당(다양성 레이어).
- 게임플레이 구조(이벤트, 입출구)는 고정 → 밸런스·서사 흐름 유지
- 일반 방의 타일 조합으로 전투·탐색 다양성

**③ 시작 방 중심 확장 방식 (The Binding of Isaac)**
시작 방에서 사방으로 방을 붙여가며 확장. 막다른 방(연결 없음) → 보스/보물방, 인접 방이 많은 위치 → 비밀방으로 자동 지정(보장 레이어). 확장 형태 자체는 무작위(다양성 레이어).
- 특수 방 위치가 지형 조건으로 자연스럽게 결정 → 수작업 배치 없이 밸런스 보장
- 확장 형태가 매번 달라 맵 위상이 전혀 다름

**설계 판단 기준**
- 적합: 로그라이크/로그라이트 — 반복 플레이가 전제이므로 다양성 가치가 높음
- 비적합: 서사 중심 게임 — 개발자 의도 배치가 중요한 경우 절차 생성이 서사 밀도를 희석
- 혼합 가능: 오픈 월드 지형은 절차 생성 + 퀘스트/던전 내부는 수작업

대표 사례:
- **Spelunky** — 바둑판 + 경로 우선 + 템플릿
- **Dead Cells** — 지역 고정 + 타일 세트 교체
- **The Binding of Isaac** — 시작점 확장 + 위상 기반 특수 방 배치
- **Enter the Gungeon** — Isaac과 유사한 확장 방식 + 방 형태 다양화

## 4. 동적 난이도 조절 (DDA, Dynamic Difficulty Adjustment)

_플레이어의 실력·상태를 실시간으로 관찰해 난이도를 자동 조정하는 시스템. 지루함(너무 쉬움)과 좌절(너무 어려움) 사이의 플로우 존(Flow Zone)을 유지하는 것이 목적._

**개념**
플레이어의 퍼포먼스(사망 횟수, 진행 속도, 점수 등)를 실시간으로 측정하고, 그 결과에 따라 적 강도·아이템·장애물을 자동 조정하는 시스템.

**핵심 원리**
고정 난이도는 숙련자에게 쉽고 초보자에게 어렵다 — 어느 한쪽이 이탈한다. DDA는 항상 "적당한 도전" 상태를 유지해 플로우 존을 사수한다.

**3가지 구현 패턴**

**① 적 강도 직접 조정 (Resident Evil 4 / Biohazard 2)**
플레이어의 피격 횟수, 사망 횟수, 진행 속도를 수치화. 퍼포먼스가 낮으면 적 체력·공격력·스폰 수를 줄이고, 높으면 늘린다.
- 조정이 자연스러워 플레이어가 눈치채기 어려움
- 난이도 변화가 점진적이라 몰입 유지에 유리

**② 러버 밴딩 (Rubber Banding) — 마리오 카트**
플레이어와 CPU 간 격차를 고무줄처럼 강제로 당기는 방식:
- 플레이어가 뒤처질 때: CPU 속도 ↓ + 강력한 따라잡기 아이템 지급
- 플레이어가 앞설 때: 특정 CPU를 "라이벌"로 지정, 코스·장애물 무시하고 비정상 속도로 추격
- **효과**: 순위와 무관하게 항상 긴장감 있는 엎치락뒤치락 레이스 연출
- **부작용**: 잘 달려도 갑자기 추격당하는 느낌 → 숙련자에게 인위적으로 느껴짐

**③ 사망 구제 시스템 — 크래시 밴디쿳**
같은 지점에서 반복 사망 시 난이도를 단계적으로 완화:
- 장애물 이동 속도 감소
- 원래 맵에 없던 중간 체크포인트·아이템 즉석 생성
- **효과**: 초보 플레이어의 중도 이탈 방지, 좌절 없이 진행 가능
- **부작용**: 반복 사망을 유도하면 아이템을 얻을 수 있다는 역이용 가능성

**장단점**

| | 내용 |
|---|---|
| 장점 | 숙련도와 무관하게 게임 완주 가능 → 리텐션 ↑, 이탈률 ↓ |
| 단점 | 시스템을 인지한 숙련자에게 "실력이 무의미하다"는 불만 → 경쟁 요소가 강한 장르에서 역효과 |

**설계 판단 기준**
- 적합: 서사 중심 싱글플레이어, 캐주얼, 레이싱 (마리오 카트처럼 재미가 경쟁보다 우선)
- 비적합: 공정한 경쟁이 핵심인 PvP, 실력 기반 소울류, 랭크 게임 — 인위적 조정이 게임의 가치를 훼손

대표 사례:
- **Resident Evil 4** — 적 체력·공격력 실시간 조정 (DDA의 교과서)
- **Biohazard 2** — 퍼포먼스 기반 난이도 시프트
- **Mario Kart** 시리즈 — 러버 밴딩 + 아이템 격차 보정
- **Crash Bandicoot** — 반복 사망 시 체크포인트·아이템 생성

## 5. 히트박스 설계 & 히트스톱 (Hitbox & Hitstop)

_캐릭터 외형 대신 단순 도형으로 충돌을 판정(히트박스)하되, 공격/피격을 플레이어에게 유리하게 비대칭 설계. 명중 순간 프레임을 의도적으로 멈춰(히트스톱) 타격감을 강화하는 디자인 패턴._

**개념**
캐릭터 메시 그대로 충돌 연산을 하면 연산 비용이 폭발한다. 대신 박스·구체 같은 단순 도형을 씌워 충돌을 계산한다 — 이것이 히트박스. 내 캐릭터의 **공격 히트박스**와 적의 **피격 히트박스**가 겹치는 순간 명중 판정.

**핵심 원리**
겉보기에 닿지 않았어도 억울함을 느끼게 하면 플레이어가 이탈한다. 대부분의 게임은 판정을 의도적으로 비대칭으로 설계한다:

| 판정 방향 | 설계 방향 | 이유 |
|---|---|---|
| 플레이어 → 적 (공격) | **관대하게 (넓게)** | 아슬아슬하게 때린 것도 인정 → 억울함 제거 |
| 적 → 플레이어 (피격) | **엄격하게 (좁게, 지연 활성화)** | 눈으로 보고 피할 시간 확보 → 공정하다는 느낌 |

구체 적용:
- **공격 히트박스**: 시각적 도달 범위보다 약간 넓게 설정
- **피격 히트박스 지연 활성화**: 적 공격 애니메이션이 시작되자마자 판정이 생기지 않고, 모션이 눈에 보인 뒤 일정 시간 후 활성화 → 반응 시간 확보
- **원거리 투사체**: 시각적 크기보다 히트박스를 작게 설정 → 빠른 투사체도 피할 수 있다는 느낌
- **동적 히트박스**: 이동 중 히트박스 축소, 회피 동작 중 히트박스 비활성화 (Geometry Dash 등) → 아슬아슬한 통과로 짜릿한 조작감 연출

히트박스 비대칭 설계와 히트스톱은 모두 같은 목표 — *플레이어가 공정하다고 느끼게 만드는 것*. 실제 물리 정확도보다 "공정하다는 인지"가 게임필의 핵심. [[game-design-notes]] #1 관용성 디자인의 판정 영역 적용.

**히트스톱 (Hitstop)**
공격이 명중하는 순간 화면·캐릭터 움직임을 수 프레임(보통 3~8프레임) 의도적으로 정지시키는 기법.

효과:
- "지금 정확히 맞았다"는 시각적·감각적 확신 → 타격감(임팩트) 강화
- 판정이 공정하다는 느낌 — 명중 사실이 명시적으로 전달되므로 "판정이 이상하다"는 불만 감소
- 적도 같이 정지하므로 적의 다음 행동을 읽을 여유 추가 확보

히트스톱이 없으면: 공격이 통과한 것처럼 보이거나, 피해가 발생해도 느낌이 없어 "이게 맞은 건가?" 혼란.

**설계 판단 기준**
- 히트박스 비대칭: 장르 무관하게 적용. 비적용 영역은 PvP 격투 게임의 정밀 판정 — 공격 히트박스를 넓히면 숙련자 간 오판이 발생
- 히트스톱: 타격감이 핵심인 액션/격투에 필수. 탄막 슈팅처럼 피탄 수가 많은 장르는 히트스톱이 짧거나 없음 (연출 과부하)

대표 사례:
- **격투 게임** 전반 — 프레임 단위 히트박스 설계, 히트스톱 필수
- **Geometry Dash** — 외형보다 극단적으로 작은 히트박스, 이동 중 추가 축소
- **소울류** (Elden Ring 등) — 피격 히트박스 지연 활성화로 모션 리딩 유도
- **2D 슈팅** 전반 — 탄환 히트박스를 외형의 절반 이하로 설정

## 6. 빌보드 기법 (Billboard Rendering)

_3D 공간 안에 배치된 2D 이미지가 카메라를 항상 정면으로 바라보도록 자동 회전시켜 입체감을 연출하는 렌더링 기법. 폴리곤 비용 없이 시각적 밀도를 유지하는 성능-품질 트레이드오프 결정._

**개념**
3D 메시로 모든 오브젝트를 표현하면 렌더링 비용이 폭발한다. 카메라를 항상 정면으로 바라보는 2D 이미지는 실제로는 평면이지만 입체처럼 보이는 착각을 이용해, 폴리곤 대신 텍스처로 대체하는 기법.

**핵심 원리**
"얼마나 멀리서 봐야 어색해지는가"를 기준으로 3D 메시와 빌보드 전환 거리를 결정. LOD(Level of Detail) 시스템과 결합해 근거리는 3D 메시, 일정 거리 이상은 빌보드로 자동 전환 → 씬 전체 성능 확보.

**3가지 방식**

**① Screen-Aligned Billboard (투영면 정렬)**
2D 이미지를 카메라 투영면과 완전히 평행하게 정렬. 카메라 기준 항상 동일한 각도 유지.
- 용도: 파티클, 렌즈 플레어 등 이펙트 — 화면에 귀속되어 위치보다 방향이 중요한 대상

**② View-Oriented Billboard (시점 지향)**
각 오브젝트가 카메라 위치를 개별적으로 직접 바라보도록 회전.
- Screen-Aligned 대비: 오브젝트가 공중에 붕 떠 보이는 문제 해결
- 용도: 캐릭터, 몬스터 등 실체가 있는 사물 — 시점이 바뀌어도 자연스러운 입체감

**③ Axial Billboard (축 정렬)**
Y축(세로)을 지면에 고정하고 수평 방향(Y축 회전)만 카메라를 향해 회전.
- 용도: 나무, 풀 등 지면에 고정된 오브젝트 — 상하가 흔들리면 어색해지는 대상
- 현대 게임에서도 원거리 식생(Vegetation) 렌더링에 가장 많이 쓰임

**성능-품질 트레이드오프**

| | 3D 메시 | 빌보드 |
|---|---|---|
| 렌더링 비용 | 높음 | 낮음 |
| 시각 품질 | 모든 각도 정확 | 측면·후면이 평면으로 보일 수 있음 |
| 적합 거리 | 근거리 | 중거리~원거리 |

대표 사례:
- **슈퍼마리오 64** — 나무·몬스터 일부를 빌보드로 표현 (N64 하드웨어 한계 대응)
- **현대 오픈 월드** (Skyrim, Zelda: BotW 등) — 원거리 식생은 Axial Billboard
- **파티클 시스템** 전반 — Screen-Aligned Billboard가 사실상 표준

## 7. 불운 방지 시스템 (Bad Luck Protection)

_순수 무작위(Pure Random)의 극단적 연속 불운을 시스템적으로 차단하는 확률 제어 패턴. "억울하다"는 감정이 이탈의 직접 원인이 되므로, 분산의 상한을 제어해 최악의 경험을 설계적으로 보장._

**개념**
순수 무작위는 수학적으로 연속 실패가 얼마든지 가능하다. 1% 확률을 100번 시도해도 한 번도 안 나오는 케이스가 약 37% 존재 — 통계적으로 흔한 일이지만 플레이어는 "조작당한다"고 느낀다. 불운 방지 시스템은 이 *분산의 꼬리*를 잘라낸다.

**핵심 원리**
완전 랜덤을 유지하되 분산의 상한을 제어 — 기댓값은 보존하면서 최악의 케이스만 차단.

**3가지 패턴**

**① Pity Timer / 천장 (Genshin Impact 등 가챠)**
실패할 때마다 내부 불운 게이지 누적 → 임계값 도달 시 최고 등급 확정 지급. 소프트 pity(임계값 이전부터 확률 점진 상승) + 하드 pity(천장, 강제 지급)로 2단 구성.
- **보장**: 최악의 경우(천장)가 명시적으로 정해짐 → 분산 상한 확정
- **심리**: 소프트 pity 구간에서 "곧 나올 것 같다"는 기대감 형성
- 수학적 구현 → [[math-algorithm-notes]] #6 Pity 누적 시프트

**② Shuffle Bag (Tetris 등)**
모든 종류를 한 묶음에 담아 무작위 순서로 소진. 묶음이 다 빌 때까지 다음 묶음을 열지 않음.
- **보장**: 같은 종류가 N묶음(예: 7개) 안에 반드시 1회 이상 등장
- **심리**: "저 블록이 아직 안 나왔으니 곧 나온다"는 예측 가능성 → 플레이어 전략 여지 생성
- **비용**: 구현이 단순 (Queue + Fisher-Yates shuffle)
- 오용 주의: 예측 가능성이 과도해지면 게임 깊이 ↓ (테트리스는 7개 세트라 적절한 밸런스)

**③ 유사 무작위 분포 — PRD (Pseudo-Random Distribution, Dota 2)**
실패할 때마다 다음 시도 확률을 단계적으로 증가시키고, 성공 즉시 초기 확률로 리셋.
- **핵심 설계**: 화면에 표기된 확률(예: 15%)과 실제 발동까지의 통계적 기댓값이 일치하도록 초기 확률 값을 역산해서 설정. 플레이어가 보는 숫자와 체감이 동일
- **보장**: 연속 실패 시 확률 상승으로 극단적 불운 차단. 연속 성공 가능성도 초기화로 억제
- **심리**: 확률이 오르는 것을 인지하면 "이제 터질 때가 됐다"는 기대감

**세 방식 비교**

| 방식 | 불운 차단 | 연속 성공 억제 | 예측 가능성 | 복잡도 |
|---|---|---|---|---|
| Pity Timer | 강함 (천장 확정) | 없음 | 중간 | 중간 |
| Shuffle Bag | 강함 (N회 내 보장) | 강함 | 높음 | 낮음 |
| PRD | 중간 (확률 상승) | 중간 (성공 시 리셋) | 낮음 | 높음 |

**설계 판단 기준**
- **가챠/보상계** → Pity Timer: 고가치 보상의 최악 케이스 명시 → 과금 의사결정 신뢰
- **순서 의존 콘텐츠** (퍼즐, 리듬, 전략) → Shuffle Bag: 예측 가능성이 전략 심도를 높임
- **전투 스킬 발동** → PRD: 표기 확률과 체감 일치가 중요한 실시간 전투

대표 사례:
- **Genshin Impact** — Pity Timer (소프트 74회, 하드 90회, 픽업 guarantee)
- **Tetris** — Shuffle Bag (7종 블록 1세트 단위 순환)
- **Dota 2** — PRD (Bash, Coup de Grace 등 확률형 스킬 전반)
- **Hearthstone** — Pity Timer (전설 카드 팩 보장)

## 8. Mode 7 — 2D 타일맵 스캔라인 원근 왜곡

_2D 평면 타일맵을 화면의 각 스캔라인마다 다른 배율로 아핀 변환해 3D 원근감을 흉내낸 기법. 실제 3D 렌더링 없이 하드웨어 성능 한계 안에서 레이싱 코스를 구현한 눈속임 기술._

**개념**
슈퍼 마리오 카트의 레이싱 코스는 3D 렌더링이 아니라, 2D 타일맵 이미지를 가로 줄마다 다른 배율로 늘이고 줄여서 원근감을 만들어낸 것이다. 슈퍼 패미컴(SNES) 하드웨어는 이 방식을 Mode 7이라는 이름으로 직접 지원했다.

**핵심 원리**

**아핀 변환 (Affine Transformation)**
도형 좌표에 숫자를 더하면 이동, 곱하면 확대, 각도를 적용하면 회전·기울임 — 이 기하학적 변환 전체를 아핀 변환이라 한다. 슈퍼 패미컴 하드웨어는 이 아핀 변환을 타일셋 전체에 실시간으로 적용할 수 있었다.

**CRT 스캔라인 특성 활용**
CRT(브라운관) 모니터는 위에서 아래로 한 줄씩 전자빔으로 화면을 그린다. 개발자들은 이 특성을 이용해 *한 줄을 그릴 때마다 아핀 변환 배율을 교체*하는 방식을 사용했다:
- 화면 위쪽 줄: 작은 배율 → 멀리 있는 것처럼 보임
- 화면 아래쪽 줄: 큰 배율 → 가까이 있는 것처럼 보임

결과적으로 동일한 2D 타일맵 이미지가 위쪽은 레이싱 주행 시점, 아래쪽은 미니맵 시점처럼 동시에 표시된다. 실제로는 같은 타일맵을 줄마다 다르게 왜곡한 것이다.

**구조적 한계**
가로 방향 왜곡과 확대/축소만으로 원근감을 흉내냈기 때문에, 슈퍼 마리오 카트의 모든 레이싱 코스는 높낮이가 없는 완전히 평평한 평면 구조다. 3D 눈속임의 대가로 지형 변화가 원천적으로 불가능하다.

대표 사례:
- **슈퍼 마리오 카트 (SNES)** — Mode 7로 레이싱 코스 전체를 구현
- **크로노 트리거 (SNES)** — 동일 기법으로 시계추 애니메이션 연출

## 9. Rogue 던전 생성 — 구역 분할 + DFS 미로

_3×3 구역 분할 후 구역별 무작위 방 배치 + 통로 연결, 일부 구역을 DFS 백트래킹 미로로 대체하는 알고리즘. 스펠렁키·아이작 등 현대 로그라이크 절차적 생성의 원조._

**개념**
1980년대 고전 게임 Rogue가 처음 선보인 던전 무작위 생성 방식. 매판 다른 던전을 만들기 위해 개발된 알고리즘으로, 오늘날 로그라이크 장르 전체의 시초가 된 설계.

**핵심 원리**

**방 생성 — 3×3 구역 분할**
1. 전체 지도를 3×3 = 9개 구역으로 나눈다.
2. 방을 배치할 구역을 선택 후, 각 구역 안에서 무작위 크기·위치로 방 하나를 배치한다.
3. 방들 사이를 통로(Corridor)로 연결한다.
4. 완성된 지도 위에 플레이어·계단·몬스터(알파벳)·아이템(특수 문자)을 배치한다.

구역을 먼저 나누기 때문에 방들이 겹치지 않고, 통로 연결로 클리어 경로를 보장한다 — [[game-design-notes]] #3 절차적 생성의 "보장 레이어 + 다양성 레이어" 분리 원칙의 원형.

**미로 생성 — DFS 백트래킹**
일부 구역은 방 대신 미로로 대체된다 (낮은 확률). 알고리즘:
1. 짝수 열·행 사이사이만 이동 가능한 격자를 기반으로 설정한다.
2. 무작위 시작점을 선택한다.
3. 상·하·좌·우 중 무작위 방향으로 이동하며 경로를 기록한다.
4. 막다른 길에 도달하면 이전 칸으로 되돌아가(백트래킹) 아직 가지 않은 방향을 탐색한다.
5. 모든 칸을 순회할 때까지 반복 — 이동 경로 전체가 미로가 된다.

깊이 우선 탐색(DFS)의 직접 적용. 되돌아가는 구조 덕분에 연결되지 않은 고립 구역이 발생하지 않는다.

대표 사례:
- **Rogue (1980)** — 본 알고리즘의 원조. ASCII 문자로 던전 전체를 표현
- **Spelunky** — 바둑판 + 경로 우선 방식([[game-design-notes]] #3 ①)으로 발전
- **The Binding of Isaac** — 시작점 확장 방식([[game-design-notes]] #3 ③)으로 발전

## 10. 가챠 씨네마틱 다단계 공개 연출

_연속 뽑기를 개별 연출의 반복이 아니라 단일 씨네마틱 이벤트로 묶어 드라마틱 완결성을 부여하는 구조. 수렴→개별공개→정리의 3단계로 기대감의 파고를 설계한다._

**문제: 1회권 연출 반복 방식**
아이템 수만큼 동일한 연출(룰렛→비행)을 반복 실행하면:
- 10연챠 시 같은 연출 10번 반복 → 무감각
- 연출 단위가 아이템 1개로 고정되어 전체 흐름에 클라이맥스가 없음

**해결: 3단계 씨네마틱 구조**

| 단계 | 내용 | 역할 |
|---|---|---|
| Phase 1 | 6심볼 정육각형 수렴 + 파티클 축포 | 기대감 빌드업 (전체 1회) |
| Phase 2 | 개별 Sunburst 띠 + 심볼 도장 → 아이템 크로스페이드 | 공개 드라마 (아이템별 클릭 진행) |
| Phase 3 | 결과 목록 + 확인 버튼 | 정리 및 만족감 (전체 1회) |

Phase 1이 기대감을 한번에 모으고, Phase 2에서 한 개씩 클릭으로 공개해 긴장감을 유지, Phase 3에서 전체를 한 눈에 보여 완결.

**기각 대안**
- 기존 GachaSequence 확장 → 연출 패러다임이 달라 전체 교체가 복잡도 최소화에 유리. 기각.
- 일괄 동시 공개 → 아이템별 클릭 인터랙션 없어 주목 시간 없음. 기각.

⚠ **설계 주의점**
- **Phase 2 스킵 버튼** — 강제 대기는 초조함으로 역효과. 전체 스킵 + 현재 아이템 빠른 진행 모두 제공 필요
- **Phase 1 등급 연동** — 파티클 연출(Normal/Rare/Epic)을 결과 최고 등급에 맞추면 기대감 빌드업에 정보가 담김

## 11. Z 타게팅 (Lock-on Targeting) — 3D 액션 자동 조준

_Z 버튼 하나로 카메라와 캐릭터를 동시에 가장 가까운 적에게 고정하고, 아날로그 스틱 이동 축을 적 중심으로 재매핑하는 시스템. 3D 게임에서 "어떻게 적을 때리는가" 문제를 해결한 역사적 UX 혁신._

**개념**
슈퍼 마리오 64 등 초기 3D 게임에서는 카메라를 맞추면서 동시에 적을 공격하는 것이 구조적으로 어려웠다. 젤다의 전설: 시간의 오카리나(1998)는 Z 버튼 하나로 카메라·캐릭터·공격 방향을 적에게 동시에 고정하는 Lock-on 시스템을 도입해 이 문제를 해결했다.

**핵심 원리**

**타게팅 고정**
- Z 버튼 누름: 가장 가까운 적에게 카메라·캐릭터 방향 동시 고정
- 버튼 재입력: 다음 거리 순의 적으로 타겟 이동
- Z 버튼 해제: 고정 해제

**이동 축 재매핑 (Lock-on 중)**
락온 전 아날로그 스틱은 월드 공간 기준으로 이동. 락온 활성 시 적-캐릭터 벡터를 중심축으로 재매핑:

| 스틱 | 락온 OFF | 락온 ON |
|---|---|---|
| 위·아래 | 임의 방향 이동 | 적에게 직선 접근 / 후퇴 |
| 좌·우 | 임의 방향 회전 | 적 중심으로 원형 사이드스텝 |

이동 축만 교체해도 공방 전체가 적 중심으로 자연스럽게 재구성된다.

**UX 전달 장치 — 요정 나비(Navi)**
시스템이 혁신적이어도 플레이어가 이해하지 못하면 의미가 없다. 닌텐도는 추상적인 타게팅 상태를 세계관 안에서 시각화하기 위해 요정 나비를 설계했다:
- **위치**: 타게팅된 대상 머리 위를 비행 — "지금 이 적이 잠겨 있다"는 직관적 표시
- **색상**: 대상 종류(적/NPC/오브젝트)에 따라 색상 변경 — 추가 텍스트 없이 상태 전달
- **대사**: 대상에 관한 힌트 제공 — 정보·내비게이션 기능을 1개 캐릭터로 통합

나비는 시스템 상태를 세계관 안에 자연스럽게 편입시킨 다이에제틱(diegetic) UI 설계의 초기 사례다.

**설계 판단 기준**
- 적합: 3D 액션·어드벤처 — 카메라 조작과 전투가 동시에 요구될 때 인지 부하 경감
- 비적합: 다수 동시 교전이 핵심인 장르 — 단일 타겟 고정이 상황 인식을 오히려 제한할 수 있음 (몬스터 헌터는 명시적 락온 없이 카메라 수동 관리를 선택)

대표 사례:
- **젤다의 전설: 시간의 오카리나 (1998)** — Z 타게팅 원조. 이후 젤다 시리즈 전작 계승
- **다크 소울** 시리즈 — 동일 원리(R3 락온, 좌우 스틱 타겟 전환)를 소울류 공방 설계에 적용
- **바이오하자드 4** — 레이저 포인터 + 3인칭 고정 카메라로 락온 없이 조준 문제를 다른 방식으로 해결

## 12. 화학 엔진 (Chemistry Engine) — 오픈월드 상호작용 시스템

_게임 속 오브젝트를 Element(원소)와 Material(물질)로 분류하고 단 3가지 법칙으로 무한한 환경 이벤트를 창발시키는 상태 변화 시스템. 물리 엔진과 별개 레이어로 존재하며, 젤다의 전설: 야생의 숨결의 압도적 자유도를 뒷받침하는 핵심 설계._

**개념**
물리 엔진이 충돌·중력 같은 물리 법칙을 담당한다면, 화학 엔진은 사물과 원소의 *상태 변화*를 담당한다. 불이 나무를 태우고, 물이 불을 끄고, 전기가 금속을 타고 흐르는 모든 상호작용이 화학 엔진의 3법칙으로 결정된다.

**핵심 원리**

**두 가지 분류**
| 분류 | 정의 | 예시 |
|---|---|---|
| **Element (원소)** | 형태·실체 없는 자연 원소 | 불, 물, 얼음, 전기, 바람 |
| **Material (물질)** | 물리적 실체가 있는 사물 | 나무, 바위, 플레이어, 몬스터 |

**3가지 상호작용 법칙**
1. **Element → Material**: 원소가 물질의 상태를 바꿀 수 있다. (불 + 나무 = 연소 → 숯)
2. **Element → Element**: 원소끼리 서로의 상태를 바꿀 수 있다. (불 + 물 = 소화)
3. **Material ↛ Material**: 물질끼리는 서로의 원소 상태를 바꿀 수 없다. (실체 간 충돌은 물리 엔진 담당 — 화학 상태 변화 없음)

법칙 자체는 극단적으로 단순하지만, 조합이 얽히면서 개발자가 의도하지 않은 플레이가 *창발(emergence)*한다.

**설계 판단 기준**
- **규칙이 아닌 분류가 핵심**: Element N종 × Material M종 = N×M 상호작용이 3법칙만으로 자동 생성. 개별 케이스를 하드코딩하지 않아도 된다
- **물리 엔진과 역할 분리**: 물리 엔진은 "어떻게 움직이는가", 화학 엔진은 "어떤 상태로 변하는가" — 두 레이어가 독립적으로 존재해야 복잡도를 관리할 수 있다
- **2D 프로토타입 검증 필수**: 닌텐도는 본 게임 개발 전 2D 프로토타입으로 3법칙이 기획대로 작동하는지 먼저 검증했다 — 단순해 보여도 예외 케이스와 법칙 충돌은 초기에 발견해야 한다
- **창발의 부작용**: 플레이어가 개발자 의도 밖의 방법으로 퍼즐을 풀거나 전투를 우회하는 것이 설계상 허용된다 — 자유도가 목적이라면 이것이 결함이 아닌 성공 지표

대표 사례:
- **젤다의 전설: 야생의 숨결 (2017)** — 화학 엔진의 교과서. Element 5종(불/물/얼음/전기/바람) × Material(나무/금속/풀/물 등) 조합이 3법칙으로 상호작용

## 13. 위험 대 보상 (Risk vs Reward)

_플레이어에게 안전-소보상 vs 리스크-대보상의 두 선택지를 동시에 제시하는 구조. 위험을 자발적으로 감수하고 성공했을 때의 쾌감이 게임의 핵심 동기를 형성한다._

**개념**
게임은 대부분의 주요 순간에 두 갈래 선택지를 플레이어 앞에 놓는다:

| 선택 | 특징 |
|---|---|
| **안전 경로** | 위험 낮음, 보상 작음 |
| **위험 경로** | 위험 높음, 보상 큼 |

플레이어는 강제당하지 않는다. 더 큰 보상을 원하면 스스로 위험을 선택한다. 그 선택이 성공했을 때의 쾌감과 실패했을 때의 긴장감이 모두 게임 경험의 밀도를 만든다.

**핵심 원리**
중요한 것은 **선택의 자발성**이다. 위험이 강제된다면 단순한 장애물이지만, 플레이어가 "굳이 위험을 택한다"는 능동적 결정을 내릴 때 성공 시 쾌감이 극대화된다. 쾌감의 크기는 감수한 위험의 크기에 비례한다.

**고전 게임 사례**

**슈퍼 마리오 — 적 처치 선택**
- 안전 경로: 적을 피해 달린다 → 진행은 되지만 추가 보상 없음
- 위험 경로: 적의 머리 위로 타이밍을 맞춰 점프 → 적 처치 + 점수 획득
- 위험을 감수한 처치 성공 시 쾌감이 단순 회피보다 훨씬 강하게 전달됨

**팩맨 — 파워 펠릿(Power Pellet)**
- 기본 상태: 유령에게 잡히면 생명 소모 → 안전하게 피하는 것이 원칙
- 파워 펠릿 습득: 일시 무적 + 유령을 역으로 잡아 고득점 가능
- 리스크: 무적 종료 타이밍 불명확 → 마지막 유령을 쫓다가 효과가 끝나면 즉사
- 더 많은 유령을 잡을수록 점수 증가 → 탐욕을 부추기는 설계

**현대 게임 사례**

**블러드본(Bloodborne) — 리게인 시스템 (Regain)**
- 피격으로 체력이 깎인 직후 짧은 시간 동안 적을 공격하면 잃은 체력을 회복 가능
- 안전 경로: 뒤로 물러나 물약 사용 → 안전하지만 느리고 자원 소모
- 위험 경로: 피격 직후 즉시 역공 → 체력 회복 + 공격 모멘텀 유지
- 리스크: 역공 도중 추가 피격 시 잔여 체력까지 소진되어 사망
- 효과: 수세에 몰릴수록 공격적으로 반응하도록 플레이어를 유도 → 액션의 밀도 상승

**설계 판단 기준**

**보상은 위험의 크기를 정당화해야 한다.**
위험 대비 보상이 지나치게 작으면 플레이어는 항상 안전 경로를 선택한다. 반대로 위험 경로의 보상이 압도적이면 사실상 위험 경로가 강제된다. 두 경로 모두 "선택할 이유가 있어야" 긴장감이 살아있다.

| 설계 실수 | 결과 |
|---|---|
| 위험 보상이 너무 작음 | 모든 플레이어가 안전 경로만 선택 → 시스템 무의미 |
| 위험 보상이 압도적임 | 위험 경로 사실상 강제 → 선택 자발성 소멸 |
| 위험과 보상 간 인과 불명확 | 플레이어가 선택 결과를 예측할 수 없음 → 랜덤처럼 느껴짐 |

**적용 유형별 정리**

| 유형 | 예시 | 핵심 리스크 |
|---|---|---|
| 전투 진입/회피 | 슈퍼마리오 적 밟기 | 피격 타이밍 |
| 자원 소비 타이밍 | 팩맨 파워 펠릿 | 무적 종료 불확실성 |
| 체력/자원 회복 방식 | 블러드본 리게인 | 역공 중 추가 피격 |
| 고난도 루트 선택 | 로그라이크 보너스 방 | 강적·함정 |
| 타이밍 도박 | 가챠 기다리기 vs 지금 사용 | 기회비용 |

대표 사례:
- **슈퍼 마리오** 시리즈 — 적 처치 vs 회피
- **팩맨** — 파워 펠릿 무적 타이밍 관리
- **블러드본** — 리게인 시스템
- **다크 소울** 시리즈 전반 — 에스트 회복(무방비 노출) vs 전투 유지, 파이어링크 귀환(소울 잃음) vs 탐험 지속

- **vs game-technique-notes**: "왜 그렇게 느껴지는가"(원리) ↔ "어떻게 만드는가"(구현). 같은 주제(예: 점프 관용성)라도 본 노트는 "왜 관용성이 필요한가"를, game-technique-notes는 "어떻게 코드로 구현하는가"를 다룬다.
